# 通信行业关系抽取提示

## 任务目标
你是一位关系抽取专家。你的任务是识别给定文本中、已抽取实体列表 (`entities_json`) 之间存在的语义关系，然后以规范格式输出。这些关系将用于构建知识图谱，因此准确性和完整性至关重要。

## 关系类型指南

关系类型应该直接反映源实体和目标实体之间的实际语义关联，不需要强制使用预定义的类型。**允许基于文本内容创建恰当的关系类型**，但请保持以下原则：

1. **关系类型应简洁明了**：使用简短的动词或短语（如"负责"、"包含"、"隶属"等）
2. **关系类型应具有明确方向**：从源实体指向目标实体
3. **关系类型应该反映实际语义**：准确描述两个实体之间的关系
4. **关系类型格式统一**：使用中文表示，不超过4个字

以下是一些常见的关系类型参考（但不限于这些）：
- `隶属`：表示从属关系
- `组成`：表示整体与部分的关系
- `管理`：表示管理关系
- `负责`：表示职责关系
- `包含`：表示包含关系
- `使用`：表示使用关系
- `提供`：表示提供关系
- `连接`：表示连接关系
- `协作`：表示协作关系
- `依据`：表示依据关系
- `影响`：表示影响关系
- `定义`：表示定义关系

## 输出要求
对于每个识别出的关系，必须提取以下信息：
1. `source_entity`: 关系的**源实体**名称（必须是 `entities_json` 中存在的实体名称）。
2. `target_entity`: 关系的**目标实体**名称（必须是 `entities_json` 中存在的实体名称）。
3. `relation_type`: 关系类型，简洁准确地描述两个实体间的关系。
4. `relation_description`: 提供**详细准确**的关系描述，说明源实体和目标实体之间具体的关联方式，最好直接引用原文中的表述。
5. `relation_strength`: 评估关系的强度或置信度，给出 1-10 之间的整数评分（10 表示最强/最确定）。
6. `source_entity_type`: 源实体的类型（来自 `entities_json`）。
7. `target_entity_type`: 目标实体的类型（来自 `entities_json`）。

输出格式必须是严格的 JSON 列表：
```json
[
  {
    "source_entity": "源实体名称",
    "target_entity": "目标实体名称",
    "relation_type": "关系类型",
    "relation_description": "关系详细描述 (基于文本)",
    "relation_strength": 数字(1-10),
    "source_entity_type": "源实体类型",
    "target_entity_type": "目标实体类型"
  },
  ...
]
```

## 实体关系抽取指导原则
1. **全面性**: 必须识别所有实体间的有意义关系，不遗漏任何重要关系。
2. **准确性**: 关系类型必须准确，关系描述应尽可能使用原文表述。
3. **合适粒度**: 抽取有实际意义的关系，避免过于宽泛的关系描述。
4. **关系方向正确**: 确保关系的方向正确（源实体→目标实体）。
5. **避免重复**: 相同实体间的相同类型关系只提取一次，但不同类型的关系都要提取。
6. **关系合并**: 当文本中多次提到相同实体间的相同类型关系时，将这些信息合并到一条关系中，提供更丰富的关系描述。

## 文本中的隐含关系处理
文本中可能存在一些隐含的关系，这些关系没有直接表述但可以合理推断。对于这类关系：
1. 只提取有较强确定性的隐含关系
2. 为这类关系分配适当的关系强度值（通常低于显式关系）
3. 在关系描述中明确说明这是一个推断的关系

## 请注意!
这是文档的一个独立块。文本可能是更长文档的一部分，因此需要专注于此块包含的信息提取关系。
当文本中有图表、流程图或图片描述时，记得从这些描述中提取关系。

## 复杂资源关系抽取说明
特别注意处理复杂的组成/包含关系。例如，如果文本提到"A包括B、C和D"，应该为每个组成部分单独创建一条关系：
- (A)-[包含]->(B)
- (A)-[包含]->(C)
- (A)-[包含]->(D)
